# 边缘AI防溺水平台总体架构设计（Java 端到端）

## 1. 目标与边界

### 1.1 目标
- 在边缘盒子实现泳池防溺水实时预警，支持多视频通道、每通道不同算法模型。
- 支持远程模型管理、设备管理、批量运维（重启/升级/配置下发/日志回收）。
- 支持四类事件：长时没顶、垂直挣扎、异常静止、危险区滞留。
- 告警仅上传截图/短片段，不上传完整视频。

### 1.2 非目标
- 不替代救生员，不做“100%自动判定溺水”。
- 不依赖公网持续在线；离线时必须可本地工作。

## 2. 总体架构

- **边缘侧（Edge Runtime）**
  - 视频接入层（RTSP/NVR）
  - 推理执行层（模型调度 + 规则引擎）
  - 告警中心（分级告警 + 截图/短片段）
  - 运维代理（远程命令、配置、升级、重启）
  - 本地存储（事件、索引、日志、离线队列）

- **云管控侧（Control Plane）**
  - 设备管理服务
  - 模型仓库服务
  - 配置中心
  - 运维任务编排（批量任务）
  - 告警接收与通知服务（微信/短信/Webhook）
  - 监控与审计服务

- **数据原则**
  - 原始视频留边缘，仅回传元数据 + 截图 + 短片段（如前后各5秒）。

## 3. 边缘侧服务分层（Java）

### 3.1 Process 级组件
- `edge-agent`：设备注册、心跳、任务拉取、执行器（重启/升级/配置下发）。
- `stream-gateway`：通道管理、拉流、解码、抽帧、断流重连。
- `inference-engine`：模型加载、推理调度、后处理。
- `event-engine`：四类事件规则判断、多帧确认、告警分级。
- `alert-uploader`：截图/短片段封装、断点续传、失败重试。
- `local-store`：SQLite/嵌入式DB + 本地文件系统。

### 3.2 核心设计类
- `ChannelManager`：通道生命周期（新增/删除/启停/热更新）。
- `ModelRegistry`：模型版本、通道绑定、热切换、回滚。
- `InferenceWorker`：每通道推理任务执行器。
- `RuleEngine`：事件规则DSL执行。
- `AlarmPipeline`：去重、分级、限流、上报。
- `OpsExecutor`：远程命令执行器（白名单命令）。

## 4. 关键功能设计

### 4.1 通道-模型绑定（支持每路不同模型）
- 配置结构：`channelId -> modelProfileId -> modelVersion`。
- 支持N:1（多通道共享模型实例）和1:1（独占模型）。
- 热更新流程：
  1. 拉取新模型并校验（hash/signature）。
  2. 预加载到备用实例。
  3. 通道原子切换指针。
  4. 观测窗口异常则自动回滚。

### 4.2 四类防溺水事件规则
- **长时没顶**：目标头部连续不可见 + 水域内停留 > T1。
- **垂直挣扎**：低位移 + 高频肢体运动特征 + 持续 > T2。
- **异常静止**：人体姿态异常且运动量低于阈值 > T3。
- **危险区滞留**：ROI高风险区内目标停留 > T4（儿童更高权重）。

统一规则机制：
- 多帧确认：`N帧命中 / M帧窗口`。
- ROI约束：仅在泳池有效区域和高风险区域触发。
- 多条件联合：检测分数、轨迹、时长、速度变化。
- 抖动抑制：同目标冷却窗口，避免连续刷屏。

### 4.3 告警分级
- `L1 预警`：疑似风险，值班端弹窗。
- `L2 告警`：高风险，声光 + 手机推送 + 值班确认。
- `L3 紧急`：持续高风险未处置，升级通知负责人。

### 4.4 告警载荷（仅截图/短片段）
- 截图：触发帧 + ROI框 + 事件标签。
- 短片段：`[-5s, +5s]` 或 `[-3s, +7s]`。
- 元数据：设备ID、通道ID、事件类型、置信度、持续时长、处置状态。

## 5. 远程运维与设备管理

### 5.1 设备管理
- 设备注册：首次激活（SN + 证书/Token）。
- 资产模型：设备、通道、池区、组织、责任人。
- 状态监控：在线、温度、CPU、内存、磁盘、推理FPS、掉线率。

### 5.2 远程操作（必须可批量）
- 重启服务、重启设备。
- 配置下发（通道参数、阈值、ROI）。
- 模型发布/回滚。
- 日志抓取、诊断包导出。

### 5.3 安全控制
- 命令白名单 + RBAC。
- 全链路TLS + 设备证书。
- 每次远程操作写审计日志（操作者/时间/结果）。

## 6. 数据模型（建议）

- `device`：设备基本信息、状态、版本。
- `channel`：通道地址、启停状态、池区绑定。
- `model_profile`：模型类型与默认参数。
- `channel_binding`：通道与模型版本绑定。
- `roi_region`：多边形区域、优先级、有效时段。
- `event_record`：事件明细、分级、处置结果。
- `alert_task`：上报任务、重试次数、最终状态。
- `ops_task`：运维任务、执行结果、回执。

## 7. API 设计（最小集）

### 7.1 云 -> 边
- `POST /edge/tasks`：下发任务（重启/更新配置/模型切换）。
- `POST /edge/model/publish`：发布模型版本。
- `POST /edge/channel/apply`：通道配置变更。

### 7.2 边 -> 云
- `POST /cloud/heartbeat`：心跳与资源指标。
- `POST /cloud/events`：事件元数据上报。
- `POST /cloud/artifacts`：截图/短片段上传。
- `POST /cloud/task/result`：任务执行回执。

## 8. 可靠性与可观测性

### 8.1 稳定性机制
- 拉流断线自动重连（指数退避）。
- 本地队列离线缓存（网络恢复后补传）。
- 服务守护（systemd/docker restart policy）。
- 配置和模型双版本回滚。

### 8.2 指标与SLO
- 告警端到端时延：P95 <= 10s。
- 设备在线率：>= 99.5%。
- 断流恢复时间：95% <= 10s。
- 推理实时性：单通道FPS达标（按机型定义）。

## 9. 部署与发布策略

- 分环境：`dev/staging/prod`。
- 发布策略：灰度（5%->20%->50%->100%）。
- 自动回滚条件：
  - 告警时延激增
  - 崩溃率超阈值
  - 误报率异常上升

## 10. 测试策略

- 单元测试：规则引擎、阈值判定、去重逻辑。
- 集成测试：拉流->推理->告警->上报全链路。
- 回放测试：白天/夜间/逆光/雨天样本集。
- 压测：多通道并发、网络抖动、断网恢复。

## 11. 建议的技术选型

- Java 17 + Spring Boot（控制面/边缘控制服务）。
- 视频：FFmpeg/JavaCV。
- 推理：ONNX Runtime（ARM64）或厂商NPU SDK适配层。
- 存储：SQLite（边缘）+ PostgreSQL（云）。
- 消息：MQTT（边云控制）+ HTTP(S)（文件上传）。
- 部署：Docker + systemd。

## 12. 你没提但必须补齐的点

- **处置闭环**：每条告警必须有“确认/误报/已处置”状态。
- **责任边界**：UI与合同中明确“AI辅助，不替代人工值守”。
- **时间策略**：按营业时段/清场时段切换阈值。
- **隐私合规**：最小化采集、分级权限、保留周期可配置。
- **演练模式**：支持“演练事件”不触发真实外呼。

## 13. 里程碑（8周建议）

- W1-W2：通道管理、拉流、基础推理框架。
- W3-W4：四类事件规则 + 多帧确认 + ROI。
- W5：告警分级与截图/短片段上报。
- W6：远程设备管理、模型发布、批量运维。
- W7：稳定性与观测（重连、补传、监控）。
- W8：试点联调与验收。
